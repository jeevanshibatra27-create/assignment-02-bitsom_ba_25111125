## Database Recommendation

For a healthcare startup building a patient management system, I would recommend using MySQL as the primary database. Healthcare data such as patient records, appointments, billing information, and medical history requires a very high level of consistency, accuracy, and reliability. MySQL follows the ACID properties (Atomicity, Consistency, Isolation, Durability), which ensure that transactions are processed reliably and data integrity is maintained. In healthcare systems, even a small data inconsistency could lead to serious problems, so strong transactional guarantees are essential.

MongoDB, on the other hand, follows the BASE model (Basically Available, Soft State, Eventual Consistency). This makes it more flexible and scalable for applications that deal with large volumes of unstructured or semi-structured data. MongoDB is useful when the data schema changes frequently or when horizontal scalability is required. However, for core healthcare operations where data correctness and strict consistency are critical, a relational database such as MySQL is generally the safer choice.

Considering the CAP theorem, relational databases like MySQL prioritize consistency and partition tolerance in controlled environments, which aligns well with the requirements of healthcare systems. Patient data must always remain accurate and consistent across the system.

If the company later adds a fraud detection module, the architecture could evolve into a hybrid approach. MySQL could still handle the core patient records and transactions, while MongoDB or another NoSQL database could store large volumes of logs, behavioral data, or analytical data used for fraud detection models. This combination would provide both strong consistency for critical data and scalability for advanced analytics.
